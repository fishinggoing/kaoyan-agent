# 考研择校 Agent 系统 —— 设计文档

## 1. 项目概述

为考研学生提供择校决策服务的 Agent 系统。通过多轮对话深挖用户真实需求，基于结构化的学校数据（硬指标+软信息）给出择校推荐。

**当前阶段：PoC**（命令行），验证 Agent 择校的可行性。后续扩展为全平台产品（Web / 手机 App / 桌面端）。

## 2. 系统架构

```
┌──────────────┐      ┌──────────────────┐      ┌──────────────┐
│  CLI 入口     │ ───▶ │  对话 Agent       │ ───▶ │  SQLite DB   │
│  (PoC)       │      │  (Dialogue)      │      │              │
└──────────────┘      └──────────────────┘      └──────▲───────┘
                                                        │ 写入
                                               ┌────────┴───────┐
                                               │ 数据采集 Agent  │
                                               │ (Collector)    │
                                               └────────────────┘
```

- **对话 Agent**：多轮对话，Function Calling 查询数据库，输出推荐
- **数据采集 Agent**：定时爬取、清洗、入库学校数据
- 两者通过共享数据库解耦，独立开发测试

## 3. 技术栈

| 层次 | 选择 | 理由 |
|------|------|------|
| 语言 | Python 3.12+ | AI/爬虫生态最成熟 |
| Agent 框架 | 无框架，手写 Agent Loop + Anthropic SDK | 最轻量可控 |
| 数据库 | SQLite（PoC）→ PostgreSQL（生产） | PoC 零配置，后续一行改 |
| 爬虫 | Playwright + httpx | 动态页面 + 静态接口兼顾 |
| LLM | Claude API（Anthropic SDK） | 对话 Agent 核心 |

## 4. 数据库表结构（5 张核心表）

### schools — 学校基础信息
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| name | VARCHAR(128) | 学校名称 |
| tier | VARCHAR(32) | 985 / 211 / 双一流 / 双非 |
| province | VARCHAR(32) | 所在省份 |
| city | VARCHAR(32) | 所在城市 |
| type | VARCHAR(32) | 综合/理工/师范/财经/医药等 |
| website | VARCHAR(256) | 研招网链接 |

### majors — 专业信息
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| school_id | FK→schools | 所属学校 |
| name | VARCHAR(128) | 专业名称 |
| discipline_rank | VARCHAR(8) | 学科评估等级 |
| exam_subjects | JSON/TEXT | 考试科目列表 |

### admission_scores — 复试分数线
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| major_id | FK→majors | 所属专业 |
| year | INT | 年份 |
| admission_line | INT | 复试分数线 |
| applicants | INT | 报考人数 |
| enrolled | INT | 录取人数 |
| push_free_ratio | FLOAT | 推免比例 |

### admitted_scores — 实际录取分数（对用户隐藏）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| major_id | FK→majors | 所属专业 |
| year | INT | 年份 |
| lowest_score | INT | 最低录取分 |
| avg_score | INT | 平均录取分 |
| highest_score | INT | 最高录取分 |

此表仅在用户明确询问"实际录取分数"时查询，不对普通查询结果展示。

### employment_quality — 就业质量
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| school_id | FK→schools | 所属学校 |
| year | INT | 年份 |
| employment_rate | FLOAT | 就业率 |
| avg_salary | INT | 平均薪资（如有） |
| summary | TEXT | 就业报告文字摘要 |

## 5. 对话 Agent 设计

### 5.1 两种交互模式

| 模式 | 触发 | 策略 |
|------|------|------|
| 信息咨询 | "XX大学分数线？" "计算机哪些学校好？" | 硬指标 → 软信息 → 给出看法 |
| 择校决策 | "帮我推荐学校" "我该报哪个？" | 先挖软偏好（地点）→ 确认硬条件 → 综合推荐 |

### 5.2 择校决策对话流程

1. **第 1-2 轮**：了解基本情况（专业、分数段）
2. **第 3-4 轮**：深挖软偏好（**地理位置确认是重点**）
3. **第 5-6 轮**：确认硬条件（学校层次、专业方向）
4. **最后轮**：数据查询 → 硬指标对比 → 软信息补充 → 给出建议

### 5.3 Agent 工具函数

| 工具 | 参数 | 说明 |
|------|------|------|
| `search_schools` | province, tier, type, keyword | 按条件搜索学校 |
| `get_majors` | school_id, discipline | 查专业+学科评估 |
| `query_scores` | major_id, year | 查复试线/报录比/招生人数 |
| `query_admitted_scores` | major_id, year | 查实际录取分（仅用户主动询问时调用） |
| `get_employment` | school_id | 查就业质量 |
| `compare_schools` | school_ids[], major_name | 多校横向对比 |

### 5.4 Agent 行为原则

- 至少确认地理位置偏好后才给推荐，不草率下结论
- 硬指标不带评价（"复试线350"而非"分数线不高"）
- 软信息给出分析并标明主观性
- 数据缺失/过时时直接说明

## 6. 数据采集 Agent 设计

### 6.1 数据源

| 优先级 | 数据源 | 内容 | 
|--------|--------|------|
| P0 | 研招网/学信网 | 招生简章、专业目录、国家线 |
| P0 | 各校研究生院官网 | 复试分数线、录取名单、报录比 |
| P1 | 教育部学科评估 | 学科评估等级 |
| P1 | 各校就业报告 | 就业率、薪资、去向 |
| P2 | 考研论坛/小程序 | 实际录取分数（非结构化） |

### 6.2 采集流程

定时触发（每周/每月）→ 增量判断 → 爬取（每源一个 Adapter）→ LLM 辅助清洗解析 → 写入数据库

### 6.3 PoC 简化

- 先手工录入 5-10 所学校数据跑通全流程
- 爬虫模块逐个接入，P0 优先

## 7. PoC 范围与成功标准

### 范围
- 命令行交互
- 5-10 所学校 + 3-5 个专业的手工数据
- 完整的多轮择校决策对话链路
- 信息咨询和择校决策两种模式

### 成功标准
接入真实数据源后，Agent 能处理十几个学校、不同专业的对比，给出有依据的择校推荐。

## 8. 系统边界（PoC 不做）

- 不做用户账号系统
- 不做前端 UI
- 不做实时推送/通知
- 不接支付/付费
- `admitted_scores` 表先建好但 PoC 可不填数据
